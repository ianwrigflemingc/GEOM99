DEVELOPMENT PROCESS

Successfully arriving at a desired destination takes proper planning, communication, and execution of a well laid out plan. Prior to any forward progress, a clear vision of the path to tread needs to be formed. As a group, we set out to map and create our vision. The journey to our destination is outlined below.

Deciding on a Topic

Our assignment to build an application using technologies of our choice presented the perfect opportunity to lay the groundwork and gain experience for the future design and completion of our collaborative project.

Based on the needs previously established with the Centre for Research and Innovation in the Bio-Economy (CRIBE), our team decided to refine the scope of the collaborative project down to a manageable size. The refining process took a few attempts as we considered the different useful forestry applications such as, routing, least cost path analysis, facility locations, biomass availability and Ontario forest management unit (FMU) boundaries.

In the end, our group decided to focus on creating a forestry infographic designed to present the end user with beneficial information about forestry facility locations found throughout Ontario accompanied by the Ontario FMU boundaries and its corresponding wood biomass information.

Deciding on a Platform

Our group predetermined that CRIBE uses the ESRI suite of products to run their geographic website components. The choice between which platform to pick was quickly narrowed down to Dashboards, AppBuilder and Experience Builder.

At first our group considered using Dashboards because CRIBE currently uses this application on their website. It is well suited for displaying location-based statistics on a dashboard style layout. Dashboards was quickly weeded out because we didn’t intend to incorporate statistics into our application.

AppBuilder became our next focus. It boasts 70 plus widgets that provided us with a wide array of functionality to choose from whereas Experience Builder had only 18 to pick from. Our concern with AppBuilder was it is older technology and only provided a mapcentric, fixed screen option.

Our group then turned to Experience Builder to compare the two applications. Experience Builder is new and provides options like a scrolling screen, single or multiple page options and is built of ArcGIS API 4.x for Javascript.

After considering the pros and cons of each, we decided to move forward with Experience Builder based on the fact it is newer technology.

Finding and Processing the Data

After deciding on Experience Builder as our platform, we had to find and process the data using ArcPro. Ontario GeoHub and Data Catalogue had the data we needed, consisting of (links provided):

  1. Forest Management Units (Shapefile)
  2. Ontario Wood Report (Excel file)
  3. Forest Processing Facilities (Shapefile)

Luckily the Processing Facility shapefile was ready to add to our AGOL group content as a Feature Layer. However, the other two layers required processing in ArcPro first.

The first step was to reformat the Available Wood report. This consisted of copying and pasting the rows and columns that we needed into a new excel sheet. Because the end goal was to export the excel file as a .CSV, the column headers also had to distinguish which species were merchantable vs unmerchantable.

Once the .CSV file was ready, we were able to bring it into an ArcPro workspace along with the Forest Management Units layer. The following were the steps taken in ArcPro:

  1. Join Data: Right click on the feature layer > Joins and Relates > Add Join. Once the geoprocessing tool opens: FMU layer as the Layer Name, FMU_Name as Input Joing Field, Wood Report as Join Table, and Forest Unit Name(In our case) as the matching Output Join Field
  2. Make Feature Layer: Because we wanted the convenience of having all the data in one feature class to query in Experience Builder, the joined data (temporary file) was made into a permanent feature class. First in the geoprocessing tools, click on 'Make Feature Layer' tool. Then enter the FMU layer as input features, and the output layer will generate automatically. This tool only creates a temporary (intermediate) file. This tool also allows you to decide which fields you want to carry forward, or whether you want to nix them.
  3. Feature Class to Feature Class: Next step is to export the temporary layer created in the previous step as a feature class to make it permanent using the 'Feature Class to Feature Class tool'.
  4. Share as a Web Layer: The last step was to get the new merge feature class onto the AGOL group account. There are two options:
    • We could have logged on to ArcGIS Online and added the layer on the 'My Content' web page through and then Add Item > From your computer, and finally shared to the group, or;
    • Simply Share as a Web Layer under 'Share'. This allows you to not only upload the layer to your own content, but also directly to the group folder. Much easier!

Building a Web Map

The map displayed in the final solution was configured using AGOL Map Viewer (beta). The following steps were followed to cartographically render the forest resources information.

The screen capture on the right depicts the Map Viewer Beta environment and the configuration of pup-ups for the FMU layer.

  1. Add Data: The forest processing facility point layer and the forest management unit (FMU) polygon layer containing the Ontario wood report data were added to the map. To facilitate rendering, the point layer was stacked on top of the polygon layer.
  2. Select Base Map: A simple light-gray base map was chosen to add spatial context to the map without distracting from the focus of the application.
  3. Set Symbology: The FMU layer was displayed as a solid outline to show the bounds of the management unit. Each processing facility was displayed with a unique colour based on the facility type, and then further categorised by shape to help users differentiate between the sites.
  4. Edit Fields: Many of the field names associated with the data layers were coded, so the field names for both layers were edited into user-friendly titles so that the information can be easily interpreted.
  5. Configure Pop-Ups: Fields to be included in the pop-up windows for both layers were selected based on what information was relevant to the end-user. For the facility layer we chose to include information such as the facility type, owner, identification, location, and operating status. For the FMU layer, we chose to include management information such as the FMU name, area, and licence holder, and information on the available wood resources for the area. For the FMU pop-ups, we also included a pie chart to display the total merchantable and unmerchantable timber resources in each FMU and text boxes to further explain the information displayed.

Configuring Experience Builder

  1. Select a Template: We chose to use the ESRI "Launchpad" template which has an emphasis on the map and a controller widget anchored at the bottom of the page to get us started with building our application, however we turned off the “lock layout” option so we could further customize the experience.
  2. Configure Map: The web map displaying the forest resource layers rendered in the previous task was added to the experience as the main map layer and the default view was adjusted.
  3. Update Branding: To maintain consistency with our team's branding schema, the default colours and text were updated & an image widget containing our logo was added.
  4. Configure Widgets: We chose to add five widgets to the controller widget at the bottom of the screen to add interactivity to our experience. The following widgets were configured:
    • Map Layers: The map layers widget is connected to the map and allows the user to turn on and off the FMU polygon layer and the forest processing facility point layer.
    • Legend: The legend widget displays the symbology for the two forest resource layers on the map.
    • Table: The contact information table displays the contact information for each FMU. This table has a search function which allows the user to search for information using either the FMU name or contact name.
    • Filter: The filter widget is connected to the forest processing facility point layer and allows the user to display only the selected facility types on the map.
    • Bookmark: The bookmark widget was configured to display the four main regions of Ontario. The map zooms to the pre-set area when the user selects a region from the bookmark.
